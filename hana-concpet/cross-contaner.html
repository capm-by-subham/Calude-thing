Chapter 01 · The Shape of the Problem

Two containers, one wire.

HDI containers on HANA Cloud are schema-isolated by design. You can't just SELECT * FROM other_schema.table from your app. Cross-container access is the sanctioned way to punch a hole through — one narrow, permissioned wire between exactly two containers.

In this guide, Container A (mid_night_crave-db) owns two tables: DB_USER and DB_ADDRESSES. Container B (delivery_management_system-db) is our CAP app that needs to read from them. Neither container gets modified structurally — instead, Container A publishes a couple of roles, and Container B claims them.

Container A · Provider
mid_night_crave-db
Owns the data
  • DB_USER
  • DB_ADDRESSES
  • DB_READ (role)
  • DB_READ_GRANTOR# (role)
via binding
Container B · Consumer
delivery_management_system
Reads the data
  • Synonyms → A's tables
  • CAP facade entities
  • OData services
  • Fiori UI apps
Mental model

Container A is a records room in another building. Container B is your office. The role is your key card, the synonym is a labeled file drawer in your office, the synonymconfig is the note inside saying "see file X in Room 203". You need all three, or nothing works.

Chapter 02 · The Three Files

Everything hinges on three JSON files.

Every cross-container setup, no matter how large, comes down to writing these three files in Container B's db/ folder. Each answers exactly one question.

.hdbgrants — am I allowed to see it?
Grant
Tells HDI to log into Container A and grant Container B's users the roles Container A exposed. Runs first, before any object is created. Without this, the synonym works but every query returns "insufficient privilege."
Location
db/cfg/
Plugin
di.grants
When
Before deploy
.hdbsynonym — what do I call it locally?
Synonym
Creates a synonym object in Container B's catalog. Empty by itself — just a declaration that "an object with this local name will exist here." Without this, there's no handle in Container B to reference the remote table.
Location
db/src/
Plugin
di.synonym
When
Make phase
.hdbsynonymconfig — where does the name point to?
Config
Wires the synonym to a real remote object. Contains target.object (real table name in Container A) and schema.configure (which service binding to read the schema from). This split lets you swap the target without rebuilding the synonym.
Location
db/cfg/
Plugin
di.synonym.config
When
Runtime bound
Key rule

The outer JSON key in .hdbsynonym and .hdbsynonymconfig must match exactly. That's how HDI stitches the two files together into one working synonym.

Chapter 03 · Container A · Roles

Publish two roles from the provider.

Container A defines what "read access" looks like, then hands the keys to Container B in the form of named roles. Two roles because there are two audiences with different needs.

The grantor role goes to Container B's object owner (the internal #OO user) — needs with_grant_option so it can re-share the privilege down to synonyms. The runtime role goes to Container B's application user (_RT), which is what the CAP app actually uses to query data.

mid_night_crave/db/src/DB_READ_GRANTOR.hdbrole 
{
  "role": {
    "name": "DB_READ_GRANTOR#",
    "object_privileges": [
      {
        "name": "DB_USER",
        "type": "TABLE",
        "privileges_with_grant_option": ["SELECT"]
      },
      {
        "name": "DB_ADDRESSES",
        "type": "TABLE",
        "privileges_with_grant_option": ["SELECT"]
      }
    ]
  }
}
mid_night_crave/db/src/DB_READ.hdbrole 
{
  "role": {
    "name": "DB_READ",
    "object_privileges": [
      {
        "name": "DB_USER",
        "type": "TABLE",
        "privileges": ["SELECT"]
      },
      {
        "name": "DB_ADDRESSES",
        "type": "TABLE",
        "privileges": ["SELECT"]
      }
    ]
  }
}
Naming convention

The # suffix on DB_READ_GRANTOR# is SAP convention — it marks a role that carries with_grant_option, so it's for object owners. Plain names (no suffix) are for runtime users.

After adding these files, deploy Container A first (cds deploy --to hana or cds up). The roles must exist in HANA before Container B tries to grant them to itself.

Chapter 04 · Container B · Binding

Give Container B a phone number for Container A.

Container B's project needs to know Container A's HDI service exists. The easiest way is through Business Application Studio's SAP HANA Projects view, which auto-updates mta.yaml.

Via SAP HANA Projects (BAS)

  1. Open Container B's project in BAS.
  2. Left activity bar → click the SAP HANA icon.
  3. Expand your project → Database Connections.
  4. Click + → Add HDI service instance.
  5. Pick mid_night_crave-db from the list.
  6. Accept the default binding name (cross-container-service-1) or type your own.

BAS writes two blocks to mta.yaml automatically. You don't touch them by hand.

mta.yaml (auto-added blocks) 
# In modules → db-deployer → requires:
- name: cross-container-service-1
  group: SERVICE_REPLACEMENTS
  properties:
    key: ServiceName_1
    service: ~{the-service-name}

# In resources:
- name: cross-container-service-1
  type: org.cloudfoundry.existing-service
  parameters:
    service-name: mid_night_crave-db
  properties:
    the-service-name: ${service-name}
Three names, one wire

You'll see three different names for the same connection — that trips people up. mid_night_crave-db is the real CF service. cross-container-service-1 is BAS's internal label. ServiceName_1 is the logical alias HDI uses. All three point to the same thing. See Chapter 13 for when to use which.

Chapter 05 · Container B · .hdiconfig

Register the plugins HDI will need.

HDI needs to be told which plugin handles which file type. For cross-container, that means adding hdbrole, hdbsynonym, and hdbsynonymconfig to the plugin map.

delivery_management_system/db/src/.hdiconfig 
{
  "file_suffixes": {
    "csv":              { "plugin_name": "com.sap.hana.di.tabledata.source" },
    "hdbcalculationview": { "plugin_name": "com.sap.hana.di.calculationview" },
    "hdbconstraint":    { "plugin_name": "com.sap.hana.di.constraint" },
    "hdbindex":        { "plugin_name": "com.sap.hana.di.index" },
    "hdbrole":         { "plugin_name": "com.sap.hana.di.role" },
    "hdbsynonym":      { "plugin_name": "com.sap.hana.di.synonym" },
    "hdbsynonymconfig":{ "plugin_name": "com.sap.hana.di.synonym.config" },
    "hdbtable":        { "plugin_name": "com.sap.hana.di.table" },
    "hdbtabledata":    { "plugin_name": "com.sap.hana.di.tabledata" },
    "hdbview":         { "plugin_name": "com.sap.hana.di.view" }
  }
}
Sibling folder trap

Your hand-written files live in db/src/. CAP-generated files live in db/src/gen/. HDI looks for .hdiconfig in the same folder or a parent — never sideways. So the .hdiconfig CAP generates at db/src/gen/.hdiconfig does not cover files at db/src/. You must provide your own at db/src/.hdiconfig.

Chapter 06 · Container B · .hdbgrants

Claim the roles Container A published.

This file tells HDI: reach into the service binding, log in as Container A's grantor, and give my users the two roles.

delivery_management_system/db/cfg/midnightgrant.hdbgrants 
{
  "mid_night_crave-db": {
    "object_owner": {
      "container_roles": ["DB_READ_GRANTOR#"]
    },
    "application_user": {
      "container_roles": ["DB_READ"]
    }
  }
}

Reading it line by line

  • mid_night_crave-db — the service the grants come from. Must match the CF service name (or the SERVICE_REPLACEMENTS logical name; both work).
  • object_owner — Container B's #OO user gets DB_READ_GRANTOR#, because it needs to re-grant SELECT down to the synonym it creates.
  • application_user — Container B's _RT user gets plain DB_READ, no grant option — this is what your CAP app uses at runtime.
  • container_roles — not schema_roles. This category is specific to HDI-to-HDI access. Using the wrong category is a common source of "not found" errors.
Common mistake

BAS's graphical .hdbgrants editor shows sections for Schema Roles, System Privileges, etc. — but no Container Roles section. People click "Schema Roles" thinking roles are roles. They're not. Switch to code view and use container_roles.

Chapter 07 · Container B · .hdbsynonym

Declare the local names.

In HANA Cloud, .hdbsynonym is just a declaration — empty objects. The actual target lives in the config file. Multiple synonyms can live in one file.

delivery_management_system/db/src/cross_syn.hdbsynonym 
{
  "DELIVERYASSIGNMENT_EXTERNALUSERS": {},
  "DELIVERYASSIGNMENT_ADDRESSES": {}
}

Why these specific names?

The outer key must match the DB name CAP will look for when it queries your facade entity. For an entity named ExternalUsers in namespace DeliveryAssignment, CAP derives the DB name as DeliveryAssignment_ExternalUsers. HANA folds unquoted identifiers to uppercase → DELIVERYASSIGNMENT_EXTERNALUSERS. That's what your synonym must be named.

The 8212001 lesson

If the synonym name doesn't match CAP's derived name, you get db://DELIVERYASSIGNMENT_EXTERNALUSERS is not provided by any file [8212001]. The synonym IS the provider — but if it's named DB_USER instead, CAP's generated projection can't find it. Match the names.

Chapter 08 · Container B · .hdbsynonymconfig

Point each synonym at the real remote object.

Same outer keys as the synonym file, but now with targets. This is where local names get translated to real Container A objects.

delivery_management_system/db/cfg/cross_syn.hdbsynonymconfig 
{
  "DELIVERYASSIGNMENT_EXTERNALUSERS": {
    "target": {
      "object": "DB_USER",
      "schema.configure": "mid_night_crave-db/schema"
    }
  },
  "DELIVERYASSIGNMENT_ADDRESSES": {
    "target": {
      "object": "DB_ADDRESSES",
      "schema.configure": "mid_night_crave-db/schema"
    }
  }
}

Reading the target

  • object — the real table name inside Container A. This is what actually gets queried.
  • schema.configure — instead of hardcoding Container A's schema GUID (C8C9CBABB6F94EDBB7252E163401B8A7), tell HDI to look it up from the service binding at runtime. Format: <service-name>/schema.

Never hardcode the schema GUID. If Container A is redeployed (new HDI instance), the GUID changes. schema.configure reads it dynamically each deploy.

Chapter 09 · CDS Facade + Service

Expose the remote table as a CAP entity.

Now that the synonym exists in Container B's schema, CAP can treat it like any other table. Declare a facade entity, tell CAP it already exists in the DB, expose it in a service.

db/model.cds 
namespace DeliveryAssignment;

// existing entities: WareHouse, Driver, Shipment, ...

@cds.persistence.exists
entity ExternalUsers {
  key ID    : String(36);
      NAME  : String(100);
      EMAIL : String(100);
}

@cds.persistence.exists
entity Addresses {
  key ID     : String(36);
      STREET : String(200);
      CITY   : String(100);
}
srv/admin-service.cds 
using {DeliveryAssignment as DA} from '../db/model';

service adminService {
  // ... other entities

  @readonly
  entity Users     as projection on DA.ExternalUsers;

  @readonly
  entity Addresses as projection on DA.Addresses;
}
The @cds.persistence.exists contract

This annotation tells the CDS compiler "don't generate a .hdbtable for this entity — it already exists". The entity's derived DB name (namespace + entity name, joined by underscore) must match the object it references. In our case, DeliveryAssignment.ExternalUsers → DELIVERYASSIGNMENT_EXTERNALUSERS → matches the synonym.

Element names (ID, NAME, EMAIL) must match the real column names in Container A's DB_USER, in the same case HANA stores them (usually uppercase for unquoted CDS-generated columns).

Chapter 10 · mta.yaml

The full deployment descriptor.

This is the deployed mta.yaml that ties every service together. The cross-container blocks are highlighted below; the rest is standard CAP scaffolding.

mta.yaml (cross-container blocks) 
# Under modules → delivery_management_system-db-deployer:
- name: delivery_management_system-db-deployer
  type: hdb
  path: gen/db
  requires:
  - name: delivery_management_system-db
    properties:
      TARGET_CONTAINER: ~{hdi-container-name}
  - name: cross-container-service-1     # ← the wire
    group: SERVICE_REPLACEMENTS
    properties:
      key: ServiceName_1
      service: ~{the-service-name}

# Under resources:
- name: cross-container-service-1
  type: org.cloudfoundry.existing-service
  parameters:
    service-name: mid_night_crave-db     # ← Container A
  properties:
    the-service-name: ${service-name}

The SERVICE_REPLACEMENTS group is what tells the HDI deployer at runtime: "when a file references ServiceName_1, use the mid_night_crave-db service binding for it."

Chapter 11 · Deploy & Verify

Wire it up.

Cross-container needs MTA deployment. cds deploy --to hana alone won't work — it doesn't read mta.yaml, so SERVICE_REPLACEMENTS never resolves.

Deploy order

Deploy Container A first So the two roles (DB_READ, DB_READ_GRANTOR#) actually exist in HANA. Container B can't be granted roles that don't exist yet.
Bind Container A to Container B Done once via BAS (Chapter 04). BAS updates mta.yaml. You don't repeat this unless you change providers.
Build & deploy Container B via MTA Run cds up — this wraps mbt build + cf deploy. HDI processes grants first, then synonym + config, then any views that use them.
terminal · deploy commands 
# One-time setup
npm i -g mbt
cf install-plugin -f multiapps
cf login

# Container A
cd mid_night_crave
cds up

# Container B
cd ../delivery_management_system
cds up

# Or manually:
cds build --production
mbt build
cf deploy mta_archives/delivery_management_system_1.0.0.mtar

Verify in HANA Database Explorer

  1. Open Container B in DB Explorer (BTP Cockpit → HDI container → Open in DB Explorer).
  2. Catalog → Synonyms → look for DELIVERYASSIGNMENT_EXTERNALUSERS.
  3. Right-click → Generate SELECT statement → Run. Rows from Container A's DB_USER should appear.
  4. Hit the OData endpoint: /odata/v4/admin-service/Users — should return the same data through CAP.
Verified

All the files in this guide were deployed successfully to BTP Trial. The chain synonym → grant → CAP projection works end-to-end. Runtime user queries flow through the synonym into Container A without either app touching the other's schema directly.

Chapter 13 · Name Glossary

The three names for the same wire.

One of the biggest sources of confusion: BAS produces three different names for what is essentially the same binding. Here's what each is and when to use it.

Name What it is Where it appears
mid_night_crave-db Container A's real Cloud Foundry service instance name. mta.yaml → service-name. Also usable directly in .hdbgrants and .hdbsynonymconfig.
cross-container-
service-1		 BAS's internal label for the binding. Auto-generated; you can rename. mta.yaml → resource name and requires block. Never in .hdb* files.
ServiceName_1 Logical alias for the binding, used by HDI at runtime. mta.yaml → SERVICE_REPLACEMENTS.key. Usable in .hdbgrants and .hdbsynonymconfig as an alternative to the real service name.
Which to use in .hdb* files

Both the real service name (mid_night_crave-db) and the SERVICE_REPLACEMENTS alias (ServiceName_1) work in .hdbgrants and .hdbsynonymconfig. The alias is more portable (change the provider by editing only mta.yaml). The real name is more direct. Pick one, stay consistent.
